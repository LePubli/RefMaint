#  OBJECTIF
Ajouter un module "GMAO Visuelle" à l'application TriMaint existante (FastAPI + React/Vite + PostgreSQL + Docker/Coolify). 
Ce module doit permettre :
1. D'afficher un schéma interactif du process de tri (image/SVG avec zones cliquables)
2. De gérer les équipements (code, nom, zone, photo, fiche technique, position sur le schéma)
3. De constituer une base de connaissances des pannes (symptômes, causes, solutions, prévention)
4. De tracer les incidents réels (historique, statut, technicien, résolution)
5. D'offrir un moteur de recherche global (équipement, panne, solution)
6. De gérer les droits : Technicien (lecture + déclaration incident) / Chef d'équipe ou Admin (CRUD complet + validation)

# 🛠️ CONTRAINTES TECHNIQUES
- Stack existante : FastAPI (backend), React+Vite+TypeScript (frontend), PostgreSQL 16, Docker Compose, Coolify
- Authentification JWT déjà en place. Ajouter un champ `role` (string: "admin", "chef_equipe", "technicien") au modèle User s'il n'existe pas.
- Stockage fichiers : `/data/uploads` (déjà monté en volume Docker). Retourner les chemins relatifs dans les réponses API.
- Respecter la structure actuelle du projet. Ne pas modifier `docker-compose.yml` sauf si strictement nécessaire.
- Générer et appliquer les migrations Alembic automatiquement.
- UI responsive, TailwindCSS déjà utilisé.

# 📋 PLAN D'EXÉCUTION SÉQUENTIEL (À suivre strictement dans l'ordre)

## ÉTAPE 1 : Backend - Modèles & Base de données
1. Dans `backend/models.py`, ajouter :
   - `Equipement` (id, code, nom, type, zone, marque, modele, position_x/y, largeur/hauteur, photo_url, description, actif, critique, created_at, updated_at)
   - `TypePanne` (id, equipement_id FK, code, titre, symptomes, causes_possibles, solutions, prevention, niveau_critique, temps_resolution_estime, tags, actif)
   - `Incident` (id, equipement_id FK, type_panne_id FK nullable, titre, description, date_debut, date_fin, duree_arret, niveau_critique, impact_production, technicien_id FK User, resolution, pieces_rechangees, cout_estime, statut: "ouvert"|"en_cours"|"resolu"|"clos")
   - `Document` (id, equipement_id FK, titre, type_document, fichier_url, fichier_type, description, version)
2. Dans `backend/schemas.py`, créer les Pydantic models pour chaque table (Create, Update, Response). Inclure les relations imbriquées optionnelles.
3. Générer la migration Alembic : `alembic revision --autogenerate -m "add_gmao_process_module"` puis `alembic upgrade head`

## ÉTAPE 2 : Backend - API & Upload
1. Créer `backend/services/upload.py` : fonction `save_upload_file(file: UploadFile, subfolder: str) -> str` qui sauvegarde dans `/data/uploads/<subfolder>/` et retourne le chemin relatif. Gérer les extensions autorisées (jpg, png, pdf, mp4).
2. Créer `backend/routers/equipements.py` : GET/POST/PUT/DELETE + endpoint `GET /search?q=...` (recherche ILIKE sur code, nom, type, zone, tags pannes).
3. Créer `backend/routers/pannes.py` : CRUD types de pannes. Endpoint `GET /equipements/{id}/pannes`.
4. Créer `backend/routers/incidents.py` : Déclaration, mise à jour statut, historique. Endpoint `GET /equipements/{id}/incidents`.
5. Créer `backend/routers/documents.py` : POST upload, GET liste, DELETE.
6. Dans `backend/main.py`, inclure tous les routers sous le préfixe `/api`.
7. Ajouter une dépendance `get_current_active_user` avec vérification de rôle pour les routes POST/PUT/DELETE (technicien peut seulement POST incidents).

## ÉTAPE 3 : Frontend - Intégration API & UI de base
1. Dans `frontend/src/api/equipements.ts`, créer les fonctions fetch avec `axios` (ou fetch) pointant vers `/api/...`. Gérer les headers JWT.
2. Créer `GlobalSearch.tsx` : input avec debounce 300ms, appel `/api/equipements/search?q=...`, affichage dropdown résultats.
3. Créer `EquipmentCard.tsx` & `EquipmentList.tsx` : grille responsive, filtres par zone/type, bouton "Voir fiche".
4. Créer `EquipmentDetail.tsx` : layout 2 colonnes (gauche: photo + infos, droite: onglets Pannes / Incidents / Documents). Utiliser `react-tabs` ou état local pour les onglets.

## ÉTAPE 4 : Frontend - Schéma Interactif
1. Créer `ProcessMap.tsx` : container relatif, image de fond (placeholder `schema-process.jpg` dans `/public`), overlay absolu.
2. Créer `EquipmentHotspot.tsx` : div positionné via `style={{ left: `${x}%`, top: `${y}%`, width: `${w}%`, height: `${h}%` }}`, hover scale, click ouvre `EquipmentDetail` en modal ou sidebar.
3. Ajouter contrôles : zoom in/out, reset view, toggle labels.
4. Lier la recherche globale au schéma : sélection d'un résultat → scroll/zoom vers le hotspot correspondant.

## ÉTAPE 5 : Formulaires & Uploads
1. `EquipmentForm.tsx` & `FaultForm.tsx` : react-hook-form + zod validation. Champ file input pour photo/documents, appel API upload dédié.
2. Afficher preview image avant soumission.
3. Gestion états loading/error/success avec `react-hot-toast`.

2. Tester : création équipement → upload photo → apparition sur liste → clic schéma → ouverture fiche.
